`jwt` や認証・認可について調べたことを書く。

`jwt` を理解するためにまずは `jwt` がなかった世界の話から始め、それから認証・認可（`OIDC`, `OAuth`）についても少し触れていきたい。

## jwt がなかった世界の認証・認可と課題
`jwt` がなかった世界ではセッション ID を発行するのが一般的だった。

1. ユーザーはユーザー ID やパスワードで認証する。
2. サーバーは認証に成功したユーザーのセッション情報を発行し、セッション DB に保存（例えば `user_id`, 時刻, 認可情報, etc）し、ID をユーザーに返す
3. ユーザーはアプリケーションの利用時にセッション ID を同時に送る
4. サーバーはセッション ID をセッション DB に問い合わせ、ユーザーが認証済みで操作が認可されているかを確認する。

このような流れだ。

しかしクラウド化が進むなど、サービスコンポーネントが増えるにつれて課題が出てきた。どのサービスもセッション ID をセッション DB に問い合わせるためレイテンシが気になる。またセッション DB の可用性も課題だ。

これらはサーバー側で状態を持つ（ステートフル）だから生じている課題なので、状態を持たない（ステートレス）ようにしようというアイディアに至る。そうすれば各々のコンポーネントが自分で認証・認可情報を検証できるようになる。

そこで、`jwt` が生まれた。

`jwt` の詳しい正体は [RFC 7519](https://datatracker.ietf.org/doc/html/rfc7519) で定義されているが、ただの ID だったセッション ID とは違い、`jwt` 自体に情報が埋め込まれている。

そして署名もあるので、各コンポーネントは鍵さえ持てば jwt を検証できる。セッション DB を持ち毎回問い合わせる必要がない。

## jwt の中身
そんな `jwt` の中身を見てみる。こんな感じの見た目をしている。

よくみると `.` により 3 つのパートに分かれていることがわかる。

1. ヘッダー
2. ペイロード
3. 署名

であり、それぞれ `Base64` 化されている。

せっかくなので `Base64` にも少し触れる。`Base64` は任意のバイナリやテキストをアルファベット・数字・記号数種類だけを使って表現できるエンコード方式だが、これがちょうど `jwt` の使われる場所と相性が良かった。

`jwt` は HTTP header や URL, Cookie などで運搬されるが、これらの場所は使える文字について厳しい制約がある。例えば `HTTP Header` は `ASCII` しか使えないし、URL はエスケープが必要な記号がある。

`jwt` を `Base64` で表現すればその問題を安全に回避できて便利だったのだ。

### ヘッダー部分
ヘッダー部分を `Base64` デコードすると以下のような情報が得られる。ちなみにこのキーと値のペアを「クレーム」と呼ぶ。

```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

`typ` は `type` のことで、この jwt のタイプが何なのかを指す。大抵の場合値は `JWT` である。

`alg` は `algorithm` のことで、この jwt の署名アルゴリズムに何を使ったのかを記す。取れる値はいろいろあるが代表的なものを並べてみる。

- `HS256`: `HMAC + SHA256`
- `ES256`: `ECDSA + SHA-256`
- `RS256`: `RSA + SHA-256`
- `none`

アルファベットと数字の組み合わせは規則性を知れば意味がわかるということ。

`HMAC` とは `Hash-based Message Authentication Code` のことで、ハッシュ関数をベースにしているメッセージ認証コードという意味。`MAC` すなわちメッセージ認証コードとは、共通鍵方式を利用してメッセージに対して認証タグを埋め込んで検証可能にするという意味を持つ。

「メッセージ認証コード」からは共通鍵方式のことを全然想起できないが、これはまだ公開鍵方式が生まれる前の世界で定義されてしまったため…な気がする。歴史的経緯なので仕方ないと考える。

`HMAC` は `MAC` をハッシュ関数で作るという意味で、その出力が 256 bit ということだ。

`E` と `R` も同じように意味を持つ。`ECDSA` と `RSA` で、それぞれ公開鍵方式で検証する。

なお `none` は署名を行なっていないことを示す。改ざんし放題なので当然非推奨。

面白いのは、`alg` は必須クレームであるにもかかわらず、サーバー側でその値を鵜呑みにするのは非推奨になっていることだ。サーバー側で `alg` をそのまま信じてしまうと、例えば `alg: none` により検証を素通りさせてしまうリスクがあるためだ。そのため、サーバー側でそもそもアルゴリズムのホワイトリストを定義し、`alg` の値を参考にして検証アルゴリズムを選択するのがよいとされる。

### ペイロード部分
ペイロード部分もデコードしてみる。

```json
{
  "sub": "1234567890",
  "name": "John Doe",
  "admin": true,
  "iat": 1516239022
}
```

ペイロード部分は事前定義された標準クレームと好きなクレームを混ぜて作ることができる、自由な部分だ。ここに必要な情報を埋め込む。

```json
{
  ...
  "takamin": "blog",
  "scope": "read",
  ...
}
```


標準クレームは以下がある。

- **iss**
- **sub**
- **aud**
- **exp**
- **nbf**
- **iat**
- **jti**

`iss` は `Issuer` の略で、この jwt の発行者を表す。
`sub` は `Subject` の略で、この jwt が持つ情報は誰・何にまつわる情報なのか、という情報の主体を表す。一般的にはユーザー ID が入るはず。
`aud` は `Audience` の略で、この jwt を受け取って処理するべきシステムを表す。
`exp` は `Expiration` の略で、この jwt の有効期限を表す。
`nbf` は `Not Before` の略で、この jwt が有効になる時間を表す。
`iat` は `Issued At` の略で、この jwt はいつ発行されたのかを表す。
`jti` は `JWT ID` の略で、この jwt の一意の ID を表す。

### 署名部分
署名部分は `Base64(Header) + "." + Base64(Payload)` したものと鍵を、`alg` で指定した方式で署名したものが入る。

jwt を受け取った `aud` が鍵（共通鍵か公開鍵）を使って署名を検証し、この `jwt` が改ざんされていないことを確認する。

## 認証と認可 / OIDC と OAuth / 各種トークン の曖昧な理解をやめる
認証と認可。`OIDC` と `OAuth`。各種トークン（`AccessToken`, `RefreshToken`, `IdToken`...）についての曖昧な理解をやめよう。

### AccessToken, RefreshToken, IdToken について
まず以下を整理する。

`AccessToken` は認可のためのトークン。jwt 形式が多い。
`RefreshToken` は AccessToken を再発行するためのトークン。
`IdToken` は認証したことを示すトークン。

`GitHub Actions` から `AWS` を操作する際によく耳にする "OIDC 連携" を使って考える。以下のような step を書いた人も多いはず。

```yaml
uses: aws-actions/configure-aws-credentials@v6
with:
    aws-region: ap-northeast-1
    role-to-assume: arn:aws:iam::123456789100:role/my-github-actions-role
```

このステップが終わると魔法のように `AWS` が使えるようになる。つまり `AWS` の認可トークンを手に入れたことになる。
認可トークンを手に入れるには認証も必要だから、なんらかの方法で認証もしている。

ざっくり流れを説明すると認証 → 認可トークン発行の流れ。まず認証をする。
`GitHub` の `OIDC` プロバイダーに `IDToken` を発行してもらう。その `IDToken` は以下のような `jwt。`

```json
{
  "iss": "https://token.actions.githubusercontent.com",
  "sub": "repo:my-org/my-repo:ref:refs/heads/main",
  "aud": "sts.amazonaws.com",
  ...
}
```

`iss`, `sub`, `aud` は先ほど勉強した。それを踏まえてみるとなるほどと思える内容。

`aud` に書かれている `AWS STS` に対して、この `IDToken` を送り、検証結果を持ってそこから `AccessKeyId`, `SecretAccessKey`, `SessionToken` を手に入れる。これは `AccessToken` に相当する一時クレデンシャル。

そのため後続で `AWS` を操作できるようになっている。

ざっくりいうとこんな流れ。

もちろん、`AWS` が `GitHub` 発行の `IDToken` を信頼している必要がある。そのため、`AWS` 側で信頼する `OIDC` プロバイダーを設定するのはそういうこと。

`IDToken` を信頼すれば設定は終わりだが、そのままだと「`GitHub` 発行の `IDToken` であれば無条件で信じる」ということになってしまう。

そこで `IAM Role` の信頼条件に、細かい `aud` の条件や `sub` の条件を書くわけだ。

全てがつながり目から鱗。


### RefreshToken は長期トークンなので結局セキュリティリスクでは？
`RefreshToken` は `AccessToken` を再発行できるトークンである。

これは利便性のためで、`AccessToken` は短命なトークンだが、発行する手順がそこそこ大変である。そこで `RefreshToken` を使えばすぐに `AccessToken` を再発行できるようにして、安全性と利便性のバランスをとった。

しかし、`RefreshToken` がそれなりに長命なトークンなのであれば、結局それは長命トークンが抱えるセキュリティリスクを抱えていることと同じでは？（AWS の API Key と同じでは？）となる。

確かに長命だが、`RefreshToken` も安全に設計されている。

1. **有効期限がある**: 長命ではあるが有効期限があるので、無期限トークンより安全である。
2. **1回限りのローテーションが多い**: `RefreshToken` はローテーションされることが多く、1 度使うと `AccessToken` と新しい `RefreshToken` が発行される。
3. **使われる場所が狭い**: `RefreshToken` は認可サーバーにしか送られず、しかも `AccessToken` が期限切れの時に初めて使われる。
使われるタイミングが限られているので、いつでもどこでも使われる長命トークンより安全性は高め。
